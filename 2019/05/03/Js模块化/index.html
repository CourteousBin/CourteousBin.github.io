Module 加载的方法

浏览器加载

传统方法

HTML 网页中, 通过 script 标签来加载

默认情况下，浏览器是同步加载 JavaScript 脚本，即渲染引擎遇到 <script> 标签就会停下来，等到执行完脚本，再继续向下渲染。如果是外部脚本，还必须加入脚本下载的时间。

如果脚本体积很大，下载和执行的时间就会很长，因此造成浏览器堵塞，用户会感觉到浏览器“卡死”了，没有任何响应。

这显然是很不好的体验，所以浏览器允许脚本异步加载，下面就是两种异步加载的语法。

defer/ async 脚本异步加载, 渲染引擎遇到这一条命令, 就会开始下载外部脚本, 但是不会等他下载和执行, 而是执行下面的命令.

defer/ async 区别:

defer 要等到渲染UI以及其他的 javascript 脚本执行后, 才会执行.

async 一旦下载完成, 渲染引擎就会中断渲染, 执行 asynv 脚本后再度执行.

一句话, defer是 渲染再执行 async 就是下载完就执行 ,

另外, 如果有多个 defer 脚本, 会按照他们在页面出现的顺序加载,

而多个 async 是不能保证加载顺序的

加载规则

浏览器加载 ES6 模块, 也可以使用 <script> 标签, 但是要加入 type="module" 属性

浏览器对于带有 type="module" 的 script 都是异步加载, 不会造成浏览器堵塞, 等到整个页面加载完成, 再执行模块脚本, 等同于 defer 属性

ES6 模块也允许内嵌在网页中, 语法与外部脚本一致

注意几点:

  • 模块代码不在全局作用域运行, 只限于自己一个模块作用域. 模块内部的顶层变量, 外部不可见
  • 模块之中, 可以使用 import / export
  • 模块之中, 顶层的 this 指向 undefined , 而不是 window
  • 同一个模块, 如果加载多次, 也只会执行一次

ES6 模块与 CommonJS 模块的差异

Node 加载 ES6 模块之前, 必须了解 ES6 和 CommonJs 模块完全不同

  • CommonJs 输出的是一个值的拷贝, ES6 模块输出的是值的引用
  • CommonJs 模块是运行时加载, ES6 模块是编译时输出接口

解释第一个差异

CommonJs 输出的是一个值的拷贝, 也就说, 不是引用类型, 一旦输出这个值, 模块内部变化不会影响到这个值.

在 main.js 加载模块

上面代码说明，lib.js模块加载以后，它的内部变化就影响不到输出的mod.counter

这是因为mod.counter是一个原始类型的值，会被缓存。

除非写成一个函数(闭包)，才能得到内部变动后的值。

ES6 模块运行机制与 CommonJS 不一样. JS 引擎对脚本静态分析的时候, 遇到模块加载命令 import , 就会生一个只读引用.

等到脚本真正执行的时候, 再根据这个只读的引用, 到加载的模块里取值.

也就是说 ES6 的 import 是动态引用, 并且不会缓存值

由于 ES6 输入的模块变量, 只是一个 “连接符” , 所以变量只读, 对他赋值就会报错

最后, export 通过接口, 输出是同一个值. 就算不同的脚本加载接口, 得到是同样的实例

Node 加载

概述

Node 对 ES6 模块的处理比较麻烦，因为它有自己的 CommonJS 模块格式，与 ES6 模块格式是不兼容的。

目前的解决方案是，将两者分开，ES6 模块和 CommonJS 采用各自的加载方案。

Node 要求 ES6 模块采用 .mjs 后缀名. 也就是说 脚本里用 import 必须是 .mjs 文件

内部变量

ES6 模块不能使用 CommonJS 特有的内部变量

  • argument
  • require
  • module
  • exports
  • __filename
  • __dirname

如果你一定要使用这些变量，有一个变通方法，就是写一个 CommonJS 模块输出这些变量，然后再用 ES6 模块加载这个 CommonJS 模块。但是这样一来，该 ES6 模块就不能直接用于浏览器环境了，所以不推荐这样做。
